URL Mapper/Router

URL <-> [Router] <-> Script

以前の書き方(CGI)

http://example.com/user.cgi?id=1

ダサい

最近の書き方

http://example.com/user/1/

いい感じ

  • URLに拡張子を含まない

  • URLに直接変数を含むことができる

  • 階層構造を簡潔に表せる

実際のコード例

Mojolicious/Perl

            
get "/user/:id" => sub {
    # Exec
}

bottle/Python

            
@get("/user/:id")
def user(id):
    # Exec

Sinatra/Ruby

            
get '/user/:id' do
    # Exec
end

Laravel/PHP

            
Route::get('user/{id}', function($id) {
    # Exec
});

WAFを用いるメリット

  • 生産性の向上

    必要な機能が(ある程度)揃っているので高速にアプリケーション開発可能に

  • バグの減少

    車輪の再開発が不要なため余計なバグが減る

  • 可読性向上

    書き方がある程度統一されるため個人差が減る

  • 大規模開発しやすい

    ドキュメントなどがあるので規約も徹底しやすい

  • セキュリティ向上

    セキュリティホールになりそうなところは隠蔽されてることも多い

WAFを用いるデメリット

  • 学習コスト

    フレームワーク毎に合わせた書き方が必要

  • 技術力が上がらない

    仕組みがわからなくても使えてしまう

    • オレオレフレームワークのすゝめ

    自分による、自分のための、自分だけのフレームワーク

    • ドキュメントなんてない

    • 安定してない

    • 設計がクソ

    • セキュリティに問題がある

    • 保守されない

    オレオレFWを作る理由

    • わからないことがわかる

      意外と書けない

    • 読解力が上がる

      大手FWではどうやって書いてるのか？

    • 設計力が上がる

      大体途中で書き直す

    おすすめな指針

    • 無計画

    • 無責任

    • 無テスト

    • 有限時間
